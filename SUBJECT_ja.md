## I. まえがき

迷路は何千年もの間、人類を魅了してきました。ギリシャ神話に登場する伝説の **クノッソスの迷宮**（ミノタウロスを閉じ込めるためにダイダロスが造った）から、現代のパズル本やビデオゲームに至るまで、迷路は常に「神秘」「挑戦」「巧妙な設計」を象徴してきました。

コンピュータサイエンスにおいて、迷路生成は単なる遊び以上のものです。それはアルゴリズム、ランダム性、グラフ理論の実用的な応用です。プリム法、クラスカル法、再帰的バックトラッカーといった有名な迷路生成アルゴリズムは、ネットワーク設計や手続き的コンテンツ生成といった現実世界の問題でも使われています。興味深いことに、**完全迷路**（任意の2点間に唯一の経路が存在する迷路）は、グラフ理論における **全域木（スパニングツリー）** と直接的に関係しています。

可視化して共有できる迷路を作ることは、コンピュータがいかにして「混沌から構造を生み出すか」を探求する素晴らしい方法であり、しかも楽しいものです。

> 「迷宮とは、迷うための場所ではなく、見出すための道である。」
> — 作者不詳

## II. AI に関する指示

### ● 背景

学習の過程で、AI はさまざまなタスクを支援できます。AI ツールのさまざまな機能と、それらがどのように作業を支援できるかを探求する時間を取ってください。ただし、常に慎重に向き合い、結果を批判的に評価してください。コード、ドキュメント、アイデア、技術的な説明のいずれであっても、あなたの質問が適切に組み立てられていたか、生成された内容が正確かどうかを完全に確信することはできません。仲間（peers）は、ミスや盲点を避けるための貴重なリソースです。

### ● 主なメッセージ

- ☛ 反復的・退屈なタスクを減らすために AI を使う。
- ☛ コーディング・非コーディングの両方でプロンプティングスキルを磨く（将来のキャリアに役立つ）。
- ☛ AI システムの仕組みを学び、よくあるリスク・バイアス・倫理的問題を予測し回避する。
- ☛ 仲間と協働しながら、技術スキルとパワースキルの両方を伸ばし続ける。
- ☛ 自分が完全に理解し、責任を持てる AI 生成コンテンツのみを使う。

### ● 学習者のルール

- AI ツールを探求し、その仕組みを理解する時間を取るべき（倫理的に使い、バイアスを減らすため）。
- プロンプトを書く前に自分の問題をよく考えるべき（より明確で詳細かつ適切な語彙を使ったプロンプトが書ける）。
- AI が生成したものは、体系的にチェック・レビュー・疑問視・テストする習慣をつけるべき。
- 常に仲間のレビューを求めるべき。自分だけの検証に頼らない。

### ● フェーズの成果

- 汎用的・分野特化型の両方のプロンプティングスキルを身につける。
- AI ツールの効果的な活用で生産性を高める。
- 計算論的思考、問題解決、適応力、協働力を引き続き強化する。

### ● コメントと例

- 試験や評価など、本当の理解を示さなければならない場面に頻繁に遭遇します。準備を怠らず、技術スキルと対人スキルの両方を伸ばし続けてください。
- 自分の推論を説明し、仲間と議論することで、しばしば理解の穴が見つかります。ピア学習を優先しましょう。
- AI ツールはあなた固有の文脈を欠き、一般的な回答を返しがちです。同じ環境を共有する仲間は、より適切で正確な洞察を与えてくれます。
- AI が「最も確からしい答え」を生成しがちなのに対し、仲間は別の視点や価値あるニュアンスを提供してくれます。彼らを品質チェックポイントとして頼りましょう。

> ✓ **良い実践:**
> AI に「ソート関数のテスト方法は?」と尋ねる。いくつかアイデアをもらう。実際に試して結果を仲間とレビューし、一緒にアプローチを洗練する。

## III. 共通の指示

### III.1 一般ルール

- プロジェクトは **Python 3.10 以降** で書くこと。
- **flake8** のコーディング規約に準拠すること。
- 関数はクラッシュを避けるため、例外を適切に処理すること。`try-except` ブロックで潜在的なエラーを管理する。ファイルや接続などのリソースには、自動クリーンアップのためコンテキストマネージャを優先する。レビュー中に未処理の例外でプログラムがクラッシュした場合、**動作しないもの（non-functional）とみなされる**。
- すべてのリソース（ファイルハンドル、ネットワーク接続など）はリークを防ぐため適切に管理すること。可能な限りコンテキストマネージャを使う。
- 関数の引数・戻り値・変数（適用可能な箇所）には **型ヒント** を含めること（`typing` モジュールを使用）。静的型チェックには **mypy** を使う。すべての関数は mypy をエラーなく通過しなければならない。
- 関数とクラスには PEP 257 に従った **docstring** を含めること（Google または NumPy スタイルなど）。目的・引数・戻り値を文書化する。

### III.2 Makefile

共通タスクを自動化するため、プロジェクトに **Makefile** を含めること。以下のルールを必ず含めること（必須の lint は指定されたフラグを含む。より強力なチェックのため `--strict` を試すことを強く推奨）:

- **install**: pip / uv / pipx などの好きなパッケージマネージャで依存関係をインストールする。
- **run**: プロジェクトのメインスクリプトを実行する（例: Python インタプリタ経由）。
- **debug**: Python 標準のデバッガ（pdb など）でメインスクリプトをデバッグモードで実行する。
- **clean**: 一時ファイルやキャッシュ（`__pycache__`、`.mypy_cache` など）を削除し、環境をクリーンに保つ。
- **lint**: 以下のコマンドを実行する:
  ```
  flake8 .
  mypy . --warn-return-any --warn-unused-ignores --ignore-missing-imports --disallow-untyped-defs --check-untyped-defs
  ```
- **lint-strict**（任意）: 以下を実行する:
  ```
  flake8 .
  mypy . --strict
  ```

### III.3 追加ガイドライン

- プロジェクトの機能を検証するテストプログラムを作成する（提出・採点はされない）。`pytest` や `unittest` などのフレームワークを使い、エッジケースをカバーする単体テストを書く。
- Python の生成物を除外する `.gitignore` ファイルを含める。
- 開発中の依存関係の分離には、仮想環境（`venv` や `conda` など）の使用を推奨する。

## IV. 必須課題

### IV.1 概要

設定ファイルを受け取り、迷路を生成し（**完全迷路**＝入口と出口の間に唯一の経路を持つ場合もある）、それを16進数の壁表現を使ってファイルに書き出す **迷路ジェネレーター** を Python で実装します。さらに、迷路のビジュアル表示を提供し、生成ロジックを後で再利用できるようコードを整理します。

### IV.2 使い方

プログラムは以下のコマンドで実行できなければなりません:

```bash
python3 a_maze_ing.py config.txt
```

- `a_maze_ing.py` はメインプログラムファイル。**この名前を使わなければならない**。
- `config.txt` は唯一の引数。迷路生成オプションを定義するプレーンテキストファイル。ファイル名は別のものでもよい。

プログラムはすべてのエラーを適切に処理しなければなりません（不正な設定、ファイル未検出、構文エラー、実現不可能な迷路パラメータなど）。**予期せずクラッシュしてはならず**、常にユーザーに明確なエラーメッセージを提示すること。

### IV.3 設定ファイルのフォーマット

設定ファイルは1行につき1つの `KEY=VALUE` ペアを含むこと。
`#` で始まる行はコメントであり、無視しなければなりません。

以下のキーは **必須** です:

| キー | 説明 | 例 |
|------|------|-----|
| `WIDTH` | 迷路の幅（セル数） | `WIDTH=20` |
| `HEIGHT` | 迷路の高さ | `HEIGHT=15` |
| `ENTRY` | 入口の座標 (x,y) | `ENTRY=0,0` |
| `EXIT` | 出口の座標 (x,y) | `EXIT=19,14` |
| `OUTPUT_FILE` | 出力ファイル名 | `OUTPUT_FILE=maze.txt` |
| `PERFECT` | 完全迷路にするか? | `PERFECT=True` |

- 役に立つなら追加のキー（`seed`、`algorithm`、`display mode` など）を加えてもよい。
- **デフォルトの設定ファイルを Git リポジトリ内に用意すること。**

### IV.4 迷路の要件

- 迷路はランダムに生成されること。ただし **seed による再現性が必須**。
- 各セルは東西南北（North, East, South, West）の各方位に、**0〜4個の壁**を持つ。
- 迷路は **妥当（valid）** でなければならない。つまり:
  - 入口と出口が存在し、互いに異なり、迷路の範囲内にあること。
  - 構造が **完全な連結性** を保証し、孤立したセルがないこと（「42」パターンを除く。後述）。
  - 入口・出口は特定のセルなので、**外周には壁があること**。
  - 生成データは **整合的（coherent）** であること。隣り合うセルは、壁がある場合に同じ壁を共有しなければならない。例: あるセルの東側に壁があり、その壁の向こう側の隣セルの西側に壁がない、という状態は禁止。
- 迷路は **大きな開けた領域を持てない**。通路は2セルより広くできない。例えば 2×3 や 3×2 の開けた領域は可だが、**3×3 の開けた領域は不可**。
- ビジュアル表示した際（後述）、迷路には複数の **完全に閉じたセル** で描かれた、目に見える「**42**」が含まれていなければならない。
- `PERFECT` フラグが有効な場合、迷路は入口と出口の間に **ちょうど1本の経路** を持たなければならない（＝完全迷路）。

> ℹ️ 迷路のサイズが小さすぎて「42」パターンを描けない場合は、省略してもよい。その場合はコンソールにエラーメッセージを表示すること。

### IV.5 出力ファイルのフォーマット

迷路は、1セルにつき16進数1桁を使って出力ファイルに書き出すこと。各桁はどの壁が閉じているかをエンコードする:

| ビット | 方位 |
|--------|------|
| 0 (LSB) | North（北） |
| 1 | East（東） |
| 2 | South（南） |
| 3 | West（西） |

- 壁が **閉じている** とビットが `1`、**開いている** と `0`。
  - 例: `3`（2進数 `0011`）は **南と西** の壁が開いている。`A`（2進数 `1010`）は **東と西** の壁が閉じている。
- セルは行ごとに（1行1行）格納する。
- 空行を1つ挟んだ後、以下の3要素を3行で出力ファイルに挿入する:
  - 入口の座標
  - 出口の座標
  - 入口から出口への最短の妥当な経路（`N`, `E`, `S`, `W` の4文字を使用）
- すべての行は `\n` で終わること。

## V. ビジュアル表示

プログラムは、以下のいずれかを使って迷路を視覚的に表示する手段を提供しなければなりません:

- **ターミナルでの ASCII レンダリング**、または
- **MiniLibX (MLX) ライブラリ** を使ったグラフィカル表示。

ビジュアルは、壁・入口・出口・解の経路を明確に示すこと。

少なくとも以下のタスクについて、**ユーザー操作（インタラクション）** が利用できること:

- 新しい迷路を再生成して表示する。
- 入口から出口への妥当な最短経路を表示/非表示する。
- 迷路の壁の色を変更する。
- 任意: 「42」パターンを表示するための特定の色を設定する。

追加のユーザー操作を加えてもよい。

> 画像参考（原文）:
> - ターミナルでのデフォルト描画（メニュー: 1. 迷路を再生成 / 2. 経路の表示・非表示 / 3. 色をローテーション / 4. 終了）
> - 別の迷路・最短経路・壁の色を変えた表示
> - MLX を使った描画（メニュー: `1: regen; 2: path; 3: color; 4: quit`）

## VI. コード再利用性の要件

迷路生成を、**単一のクラス**（例: `MazeGenerator`）として、将来のプロジェクトでインポート可能な **スタンドアロンのモジュール** 内に実装しなければなりません。

以下の方法を説明する短いドキュメントを提供すること:

- ジェネレーターのインスタンス化と使い方（最低限、基本的な例を含む）。
- カスタムパラメータの渡し方（サイズ、seed など）。
- 生成された構造へのアクセス方法、および少なくとも1つの解へのアクセス方法。

> ℹ️ 迷路ジェネレーターモジュールは迷路構造へのアクセスを提供するが、それは出力ファイルと必ずしも同じフォーマットである必要はない。

この再利用可能なモジュール（コードとドキュメント）全体は、後で **pip でインストール可能** な単一ファイルとして用意しなければなりません。

- このパッケージは **`mazegen-*`** という名前にし、ファイルは **Git リポジトリのルート** に配置すること。
- `.tar.gz` と `.whl` の両拡張子が許可される（標準的な Python パッケージビルドで生成されるもの）。
- フルファイル名の例: `mazegen-1.0.0-py3-none-any.whl`

パッケージをビルドするために必要なすべての要素を Git リポジトリに用意すること。これは評価時に要求されます。仮想環境（または同等の環境）で必要なツールをインストールし、ソースからパッケージを再ビルドします。

メインの **README.md**（再利用モジュールの一部ではない）にも、この短いドキュメントを含めること。

## VII. README の要件

Git リポジトリのルートに **README.md** ファイルを用意しなければなりません。その目的は、プロジェクトに馴染みのない人（仲間・スタッフ・採用担当者など）が、プロジェクトが何であるか、どう実行するか、トピックの詳細をどこで見つけられるかを素早く理解できるようにすることです。

README.md には少なくとも以下を含めること:

- **最初の1行は斜体**で、次のように書くこと:
  *This project has been created as part of the 42 curriculum by `<login1>`[, `<login2>`[, `<login3>`[...]]].*
- **「Description（説明）」** セクション: プロジェクトの目的と簡単な概要を含め、明確に提示する。
- **「Instructions（手順）」** セクション: コンパイル・インストール・実行に関する関連情報を含む。
- **「Resources（リソース）」** セクション: トピックに関する古典的な参考文献（ドキュメント、記事、チュートリアルなど）の一覧、および **AI をどう使ったか**（どのタスク・どの部分に使ったか）の説明。

> ➠ プロジェクトに応じて追加セクションが必要になる場合がある（使用例、機能一覧、技術的選択など）。

このプロジェクトで **必須となる追加項目**:

- 設定ファイルの完全な構造とフォーマット。
- 選択した迷路生成アルゴリズム。
- なぜそのアルゴリズムを選んだか。
- コードのどの部分が再利用可能で、どう使うか。
- **チームとプロジェクト管理** について:
  - 各メンバーの役割。
  - 当初の計画と、それが最後までどう変化したか。
  - うまくいったこと・改善できること。
  - 特定のツールを使ったか? どれを使ったか?

高度な機能（複数アルゴリズム、表示オプション）を実装した場合は、この README.md に記述すること。

> ℹ️ 英語を推奨。あるいはキャンパスの主要言語を使ってもよい。

## IX. 提出とピア評価

通常通り、課題を Git リポジトリに提出してください。**ディフェンス（評価）ではリポジトリ内の作業のみが評価されます。** ファイル名が正しいか念入りに再確認してください。

評価中、**プロジェクトの軽微な変更**を求められることがあります。これは、小さな挙動の変更、数行のコードの記述・書き直し、簡単に追加できる機能などです。

このステップはすべてのプロジェクトに適用されるわけではありませんが、評価ガイドラインに記載されている場合は対応できるよう準備しておく必要があります。

このステップは、プロジェクトの特定部分に対するあなたの **実際の理解** を確認するためのものです。変更は好きな開発環境（普段の環境など）で行ってよく、特定の制限時間が定められていない限り、数分で実現可能なはずです。

例えば、関数やスクリプトの小さな更新、表示の変更、新しい情報を格納するためのデータ構造の調整などを求められることがあります。

詳細（範囲・対象など）は **評価ガイドライン** で指定され、同じプロジェクトでも評価ごとに異なる場合があります。
