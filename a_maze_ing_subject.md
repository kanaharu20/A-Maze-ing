# A-Maze-ing（課題サマリー）

- 「This is the way」— 自分だけの迷路ジェネレーターを作り、その結果を表示する課題。Version: 2.1
- Python 3.10以降で迷路を生成し、16進数の壁表現で出力ファイルに書き出す
- ビジュアル表示（ターミナルASCIIまたはMLX）と、再利用可能なモジュール（pipパッケージ）化が必須

---

## Chapter IV — Mandatory part（必須パート）

### IV.1 Summary（概要）

設定ファイルを受け取り、迷路（場合によってはperfect、すなわち入口と出口の間の経路が1本だけの迷路）を生成し、16進数の壁表現を使ってファイルに書き出す迷路ジェネレーターをPythonで実装する。また、迷路のビジュアル表現を提供し、生成ロジックが後で再利用できるようにコードを構成する。

### IV.2 Usage（使い方）

プログラムは以下のコマンドで実行されなければならない：

```
python3 a_maze_ing.py config.txt
```

- `a_maze_ing.py` がメインプログラムファイルである。この名前を使わなければならない。
- `config.txt` が唯一の引数である。迷路生成オプションを定義するプレーンテキストファイルである。別のファイル名を使ってもよい。

プログラムはすべてのエラーを適切に（gracefully）処理しなければならない：無効な設定、ファイルが見つからない、構文エラー、実現不可能な迷路パラメータなど。予期せずクラッシュしてはならず、常にユーザーに明確なエラーメッセージを提供しなければならない。

### IV.3 Configuration file format（設定ファイルの形式）

設定ファイルは1行につき1つの `KEY=VALUE` ペアを含まなければならない。

`#` で始まる行はコメントであり、無視しなければならない。
以下のキーは**必須（mandatory）**である：

| Key | 説明 | 例 |
|---|---|---|
| `WIDTH` | 迷路の幅（セル数） | `WIDTH=20` |
| `HEIGHT` | 迷路の高さ | `HEIGHT=15` |
| `ENTRY` | 入口の座標 (x,y) | `ENTRY=0,0` |
| `EXIT` | 出口の座標 (x,y) | `EXIT=19,14` |
| `OUTPUT_FILE` | 出力ファイル名 | `OUTPUT_FILE=maze.txt` |
| `PERFECT` | 完全迷路（perfect maze）にするか？ | `PERFECT=True` |

有用であれば追加のキー（例：seed、algorithm、display mode）を追加してもよい。

デフォルトの設定ファイルを Git リポジトリに含めておかなければならない。

### IV.4 Maze Requirements（迷路の要件）

- 迷路はランダムに生成されなければならないが、シード（seed）による再現性が必須である。
- 迷路の各セルは 0〜4 個の壁を持ち、それぞれ東西南北（North, Est, South, West）の方位にある。
  ⚠️ 原文: "at each cardinal point (North, Est, South, West)"（"Est" は原文ママ、East の誤記と思われる）
- 迷路は有効（valid）でなければならない。すなわち：
  - 入口と出口が存在し、互いに異なり、迷路の範囲内にある。
  - 構造が完全な連結性（full connectivity）を保証し、孤立したセルがない（「42」パターンを除く。下記参照）。
  - 入口と出口は特定のセルであるため、外周には壁がなければならない。
  - 生成されたデータは一貫していなければならない：隣接するセル同士は、壁がある場合は同じ壁を共有しなければならない。例えば、最初のセルの東側に壁があるのに、その壁の向こうの2番目のセルの西側に壁がない、という状態は禁止である。
- 迷路に大きな空き領域があってはならない。通路（corridor）の幅は 2 セルを超えてはならない。
  例えば、2x3 や 3x2 の空き領域は許されるが、3x3 の空き領域は決して許されない。
- ビジュアル表示したとき（下記参照）、迷路には、完全に閉じた複数のセルで描かれた目に見える「42」が含まれていなければならない。
- `PERFECT` フラグが有効な場合、迷路は入口と出口の間に**ちょうど1本の経路**を含まなければならない（すなわち、perfect maze でなければならない）。

> ℹ️ **情報:**
> 迷路のサイズが許さない場合（つまり小さすぎる場合）、「42」パターンは省略してよい。その場合はコンソールにエラーメッセージを表示すること。

### IV.5 Output File Format（出力ファイルの形式）

迷路は、1セルにつき1桁の16進数を使って出力ファイルに書き込まなければならない。各桁はどの壁が閉じているかをエンコードする：

| Bit | 方向 |
|---|---|
| 0 (LSB) | North |
| 1 | East |
| 2 | South |
| 3 | West |

- 壁が閉じている場合はビットを 1 にセットし、開いている場合は 0 とする。
  例：`3`（2進数 `0011`）は**南**と**西**の壁が開いていることを意味する。また `A`（2進数 `1010`）は**東**と**西**の壁が閉じていることを意味する。
- セルは行ごとに格納し、1行につき1行で書く。
- 空行の後、以下の 3 要素を 3 行で出力ファイルに挿入する：
  - 入口の座標、出口の座標、そして入口から出口までの最短の有効経路（`N`、`E`、`S`、`W` の4文字を使用）。
- すべての行は `\n` で終わる。

## Chapter IX — Submission and peer-evaluation（提出とピア評価）

通常どおり、課題を `Git` リポジトリに提出すること。ディフェンス中はリポジトリ内の成果物のみが評価される。ファイル名が正しいか、遠慮なくダブルチェックすること。

評価中に、プロジェクトの簡単な**修正（modification of the project）**を求められることがある。これは、軽微な動作の変更、数行のコードの記述や書き直し、追加が容易な機能の実装などである。

このステップは**すべてのプロジェクトに適用されるとは限らない**が、評価ガイドラインに記載されている場合は準備しておかなければならない。

このステップは、プロジェクトの特定の部分に対する実際の理解度を確認するためのものである。修正は自分の選んだ任意の開発環境（例：普段のセットアップ）で行うことができ、数分以内に実行可能であるべきである——評価の一部として特定の時間枠が定められている場合を除く。

例えば、関数やスクリプトへの小さな更新、表示の変更、新しい情報を保存するためのデータ構造の調整などを求められる可能性がある。

詳細（範囲、対象など）は**評価ガイドライン（evaluation guidelines）**で指定され、同じプロジェクトでも評価ごとに異なる場合がある。

## ✅ 必須要件チェックリスト（Mandatory part）

- [ ] Python 3.10 以降で実装
- [ ] flake8 準拠
- [ ] 例外処理（try-except、コンテキストマネージャー）。未処理例外でのクラッシュは non-functional 扱い
- [ ] リソース管理（リーク防止、コンテキストマネージャー推奨）
- [ ] 型ヒント（typing モジュール）＋ mypy をエラーなしでパス
- [ ] PEP 257 準拠の docstring（関数・クラス）
- [ ] Makefile（install / run / debug / clean / lint ルール。lint-strict は任意）
- [ ] `.gitignore` を含める
- [ ] テストプログラム作成（pytest / unittest。提出・採点対象外）
- [ ] メインファイル名は `a_maze_ing.py`、実行は `python3 a_maze_ing.py config.txt`
- [ ] 引数は設定ファイル1つのみ
- [ ] 全エラーの graceful な処理＋明確なエラーメッセージ（クラッシュ禁止）
- [ ] 設定ファイル：`KEY=VALUE` 形式、`#` 行はコメントとして無視
- [ ] 必須キー：`WIDTH` / `HEIGHT` / `ENTRY` / `EXIT` / `OUTPUT_FILE` / `PERFECT`
- [ ] デフォルト設定ファイルをリポジトリに含める
- [ ] ランダム生成＋シードによる再現性
- [ ] 迷路の有効性：入口・出口が異なり範囲内／完全連結・孤立セルなし（「42」を除く）／外周に壁／隣接セルの壁の整合性
- [ ] 通路の幅は2セルまで（3x3の空き領域は禁止）
- [ ] ビジュアル表示時に完全に閉じたセルで「42」を描く（サイズ不足なら省略可＋コンソールにエラーメッセージ）
- [ ] `PERFECT=True` なら入口〜出口の経路はちょうど1本
- [ ] 出力ファイル：1セル＝16進1桁（bit0=North, bit1=East, bit2=South, bit3=West、閉=1）、行ごとに1行、空行の後に入口座標・出口座標・最短経路（N/E/S/W）の3行、全行 `\n` 終端
- [ ] ビジュアル表示：ターミナルASCII または MLX。壁・入口・出口・解答経路を明示
- [ ] ユーザーインタラクション：迷路の再生成／最短経路の表示・非表示／壁の色変更（「42」用の色設定は任意）
- [ ] 迷路生成を単一クラス（例：MazeGenerator）のスタンドアロンモジュールとして実装
- [ ] モジュールの短いドキュメント（インスタンス化と使用例／カスタムパラメータ／構造と解へのアクセス）
- [ ] `mazegen-*` パッケージ（.tar.gz または .whl）をリポジトリのルートに配置
- [ ] パッケージをソースから再ビルドできる要素一式をリポジトリに含める（評価で実演）
- [ ] README.md（共通セクション＋設定ファイル形式・アルゴリズムと選定理由・再利用可能部分・チーム/プロジェクト管理の記述）

## 🚫 禁止事項リスト（Forbidden functions/behaviors）

- 未処理の例外によるクラッシュ（non-functional 扱い）
- mypy エラーの残存
- メインファイル名の変更（`a_maze_ing.py` 必須）
- 3x3 以上の空き領域（通路幅は2セルまで）
- 隣接セル間で壁が共有されていない不整合なデータ
- `PERFECT=True` 時の複数経路
- シードなしの再現不可能な生成

## 📦 提出物リスト（Turn-in files）

- `a_maze_ing.py`（メインプログラム）
- 迷路生成モジュール（単一クラス、スタンドアロン）
- `mazegen-*` パッケージファイル（`.tar.gz` または `.whl`、リポジトリのルート）
- パッケージのビルドに必要なすべての要素
- デフォルトの設定ファイル
- `Makefile`（install / run / debug / clean / lint、任意で lint-strict）
- `.gitignore`
- `README.md`（必須セクション＋プロジェクト固有の追記事項＋再利用モジュールのドキュメント）

## ⭐ ボーナス要件

- 複数の迷路生成アルゴリズムのサポート
- 迷路生成中のアニメーション
